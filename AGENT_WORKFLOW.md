# 智能体协作开发工作流

本文档说明如何使用两个智能体（Code Generator + Code Reviewer）来开发和维护 C/C++ Config Parser 扩展。

## 智能体角色

### 🔨 Code Generator Agent

**职责**：
- 根据需求生成 TypeScript 模块代码
- 实现 .config 文件解析逻辑
- 实现 VSCode 配置管理 API 调用
- 编写单元测试

**使用时机**：
- 新增功能开发
- Bug 修复代码生成
- 测试用例编写

### 👁️ Code Reviewer Agent

**职责**：
- 检查解析逻辑的正确性和鲁棒性
- 验证 VSCode API 使用是否规范
- 检查类型安全和错误处理
- 评估性能和安全性
- 检查测试覆盖率

**使用时机**：
- Generator 完成代码后
- 重大功能变更后
- 发布前的质量把关

## 工作流程

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           开发工作流                                          │
└─────────────────────────────────────────────────────────────────────────────┘

Phase 1: 需求分析与设计
├── 开发者明确功能需求
└── Checkpoint 1: 确认功能规格和架构设计

Phase 2: Code Generator 编码
├── 调用 generator-agent 生成代码
│   └── Agent 分析需求并生成代码模块
└── Checkpoint 2: 开发者审查生成的代码
    ├── 代码结构是否符合预期
    ├── 功能逻辑是否正确
    └── 选择: [确认继续] [要求修改] [取消]

Phase 3: Code Reviewer 审核
├── 调用 reviewer-agent 审核代码
│   └── Agent 进行全面代码审查
└── Checkpoint 3: 开发者审查审核报告
    ├── 检查问题列表
    ├── 评估风险等级
    └── 选择: [通过并应用] [返回修改] [取消]

Phase 4: 测试验证
├── 运行单元测试
├── 运行集成测试
└── Checkpoint 4: 确认功能符合预期
    └── 选择: [通过] [修复问题]

Phase 5: 交付
└── 代码合并到主分支
```

## 使用示例

### 场景：新增配置格式支持

假设需要支持 autoconf.h 头文件格式（除了 .config 格式）。

**Step 1: 调用 Code Generator**

```
提示词：
为 C/C++ Config Parser 扩展新增 autoconf.h 头文件解析支持。

需求：
1. 支持解析 autoconf.h 格式的宏定义：
   #define CONFIG_FEATURE_A 1
   #define CONFIG_FEATURE_B 0
   #define CONFIG_VALUE 123
   #define CONFIG_STRING "hello"

2. 更新 ConfigParser 类，添加 parseAutoconfHeader() 方法
3. 更新配置管理器，支持自动检测文件格式
4. 添加相应的单元测试

参考：
- src/core/configParser.ts 现有实现
- src/core/configManager.ts 现有实现
- 使用相同的错误处理和日志模式

请生成完整的实现代码。
```

**Checkpoint 2: 开发者审查**
- 检查生成的代码结构
- 确认解析逻辑正确
- 确认测试覆盖充分
- 确认后继续

**Step 3: 调用 Code Reviewer**

```
提示词：
审核以下新增的 autoconf.h 解析功能代码：

代码位置：
- src/core/configParser.ts (新增方法)
- src/core/configManager.ts (格式检测逻辑)
- src/test/suite/configParser.test.ts (新增测试)

请检查：
1. 解析逻辑的正确性和鲁棒性
2. 错误处理是否完善
3. 与现有代码的集成是否顺畅
4. 测试覆盖率是否充分
5. 是否有性能或安全问题

输出审核报告，包含：
- 总体评分 (1-10)
- 发现的问题列表（严重/一般/建议）
- 改进建议
```

**Checkpoint 3: 处理审核报告**
- 阅读审核报告
- 处理严重和一般问题
- 评估建议是否采纳
- 确认通过后应用代码

**Step 4: 测试验证**
- 运行 `npm run compile` 确保编译通过
- 运行 `npm run test` 确保测试通过
- 手动验证新功能

**Checkpoint 4: 最终确认**
- 确认所有测试通过
- 确认功能符合预期
- 合并代码

## 最佳实践

### 对 Generator 的提示词建议

1. **明确需求边界**
   - 说明要实现的功能
   - 列出具体的需求点
   - 提供参考代码或示例

2. **提供上下文**
   - 相关文件路径
   - 现有实现模式
   - 需要遵循的约定

3. **指定输出要求**
   - 需要生成的文件列表
   - 测试覆盖要求
   - 错误处理要求

### 对 Reviewer 的提示词建议

1. **明确审核范围**
   - 指定需要审核的文件
   - 说明本次变更的范围
   - 指出重点关注的方面

2. **指定审核标准**
   - 正确性要求
   - 性能要求
   - 安全要求

3. **要求结构化输出**
   - 评分
   - 分级问题列表
   - 具体改进建议

## 常见问题

### Q: Generator 生成的代码不符合预期怎么办？
A: 在 Checkpoint 2 选择「要求修改」，并提供具体的修改意见。清晰的反馈会让下一次生成更接近预期。

### Q: Reviewer 提出的问题是否都要修复？
A: 严重和一般问题应该修复。建议级别的可以根据时间和优先级决定是否采纳。

### Q: 小改动是否需要完整流程？
A: 简单的修改（如文档更新、变量重命名）可以跳过 Reviewer，但建议至少自测。

### Q: 如何处理 Generator 和 Reviewer 意见不一致？
A: 以开发者的判断为准。Reviewer 可能过于保守，Generator 可能过于乐观。开发者需要根据具体情况权衡。
